    Apa yang Dimaksud Defect Clustering?

    Read More

    Defect Clustering adalah prinsip pengujian perangkat lunak yang menunjukkan bahwa sejumlah kecil modul atau fungsi dalam suatu aplikasi cenderung memiliki sebagian besar cacat. Konsep ini erat kaitannya dengan Prinsip Pareto (aturan 80/20), dimana sekitar 80% cacat ditemukan pada 20% modul.

    Poin Penting

    1. Definisi
      Konsentrasi sebagian besar cacat pada sejumlah kecil modul aplikasi.
    2. Alasan
      Modul tertentu mungkin lebih kompleks, sering diubah, atau dirancang tidak memadai, sehingga meningkatkan kemungkinan cacat.
    3. Manfaat
      1. Membantu memprioritaskan upaya pengujian pada area yang rawan cacat.
      2. Meningkatkan efisiensi proses pengujian dengan memfokuskan sumber daya secara efektif.
    4. Tantangan
      Ketergantungan yang berlebihan pada pengelompokan kerusakan dapat menyebabkan pengabaian pada area yang kurang kritis, sehingga berpotensi menghilangkan kerusakan yang tersembunyi.

    Contoh Real-time dengan Modul Pembayaran (e-Commerce Platform)

    1. Area Rawan Cacat
      Modul seperti Integrasi Payment Gateway, Aplikasi Discount Code, atau Proses Refund mungkin memiliki lebih banyak cacat karena melibatkan beberapa integrasi pihak ketiga, perhitungan, dan alur kerja yang kompleks.
    2. Area Stabil
      Bagian seperti FAQ atau Kebijakan Privasi cenderung tidak memiliki cacat karena bersifat statis dan jarang diperbarui.

    Dengan mengidentifikasi modul yang rawan cacat seperti Payment Gateway, tim dapat mengalokasikan lebih banyak sumber daya untuk menguji skenario seperti beberapa metode pembayaran, konversi mata uang, atau menangani transaksi yang gagal, sehingga memastikan keandalan sistem.

    Meningkatkan Query dengan Nesting

    Read More
    1. Apa itu Sub-query?
      Sub-kueri adalah kueri SQL bersarang yang menghitung hasil antara untuk digunakan dalam kueri utama. Contoh:

      SELECT name FROM employees WHERE department_id = (SELECT department_id FROM departments WHERE department_name = 'IT');

    2. Apa yang dimaksud dengan Co-Related Sub-query?
      Subkueri yang terkait bersama bergantung pada kueri luar, yang dijalankan untuk setiap baris secara dinamis berdasarkan konteksnya. Contoh:

      SELECT e1.name FROM employees e1 WHERE salary > (SELECT AVG(salary) FROM employees e2 WHERE e1.department_id = e2.department_id);

    3. Perbedaan Utama
      Subkueri tidak bergantung pada kueri luar, sedangkan subkueri terkait bergantung padanya, sehingga membuatnya lebih lambat untuk kumpulan data yang lebih besar.

    4. Jenis Sub-query
      Mencakup sub-kueri satu baris, multi-baris, skalar, dan bersarang, menawarkan fleksibilitas untuk berbagai kebutuhan kueri.
      1. Single Row Sub-query
        Mengembalikan nilai tunggal untuk perbandingan.
      2. Multi Row Sub-query
        Mengembalikan beberapa baris untuk dibandingkan dengan IN atau ANY.
      3. Scalar Sub-query
        Mengembalikan nilai tunggal yang dapat digunakan dalam ekspresi.
      4. Nested Sub-query
        Berisi beberapa tingkat sub-kueri.
    5. Use Cases yang Umum
      Subkueri ideal untuk pemfilteran, perbandingan, dan agregasi, sementara subkueri yang terkait bersama unggul dalam penghitungan dinamis dan pemfilteran spesifik baris. Contoh:

      SELECT name FROM employees WHERE department_id = (SELECT department_id FROM departments WHERE department_name = 'IT');

    6. Tips Kinerja
      Gunakan indeks, sederhanakan nesting, dan ganti subkueri terkait dengan gabungan atau Common Table Expressions (CTE) untuk efisiensi yang lebih baik. Contoh:

      SELECT name FROM employees WHERE department_id IN (SELECT department_id FROM departments WHERE location_id = 1);

    Pertanyaan Seputar RTM (Requirement Traceability Matrix)

    Read More
    1. Apa itu RTM?
      RTM adalah dokumen yang memetakan dan menelusuri kebutuhan pengguna dengan kasus uji untuk memastikan bahwa semua persyaratan tercakup dalam kasus uji.
    2. Mengapa RTM Penting?
      1. Memastikan cakupan tes yang lengkap.
      2. Membantu mengidentifikasi persyaratan yang hilang.
      3. Melacak kemajuan pengujian.
      4. Mendukung analisis dampak jika terjadi perubahan persyaratan.
    3. Apa saja komponen RTM?
      1. Requirement ID
      2. Requirement Description
      3. Test Case ID
      4. Test Case Description
      5. Status (Pass/Fail/Blocked)
      6. Defect ID (jika berlaku)
    4. Apa saja jenis Matriks Ketertelusuran?
      1. Forward Traceability
        Tautan persyaratan ke test case.
      2. Backward Traceability
        Menghubungkan test case ke persyaratan.
      3. Bi-directional Traceability
        Memastikan ketertelusuran ke depan dan ke belakang
    5. Bagaimana RTM dipersiapkan?
      1. Kumpulkan semua persyaratan (misalnya, dari BRD, SRS).
      2. Buat tabel dengan kolom untuk persyaratan, kasus uji, dan detail terkait.
      3. Petakan kasus uji ke persyaratan masing-masing.
      4. Alat apa yang digunakan untuk memelihara RTM?

    Pertanyaan Seputar API Testing

    Read More
    1. Apa itu API?
      Suatu cara bagi sistem perangkat lunak untuk berbicara satu sama lain.
    2. Apa keuntungan dari pengujian API?
      Temukan bug lebih cepat, uji tanpa antarmuka pengguna, rilis perangkat lunak lebih cepat.
    3. Jelaskan berbagai jenis pengujian API.
      Functional (apakah berfungsi?), Performance (apakah cepat?), Security (apakah amankah?), dan banyak lagi.
    4. Apa saja protokol umum yang digunakan dalam pengujian API?
      HTTP, REST, SOAP
    5. Apa perbedaan utama antara API dan Layanan Web?
      Semua layanan web adalah API, namun tidak semua API adalah layanan web. Layanan web menggunakan protokol web tertentu.
    6. Apa perbedaan antara pengujian API dan pengujian Unit?
      Pengujian API memeriksa sekelompok komponen, Pengujian unit memeriksa bagian-bagian individual.
    7. Apa saja komponen inti dari permintaan HTTP?
      URL, metode (seperti GET atau POST), header, dan body.
    8. Apa sajakah metode HTTP yang umum dan kegunaannya?
      GET (mengambil data), POST (mengirim data), PUT (memperbarui data), DELETE (menghapus data).
    9. Apa yang diberitahukan oleh kode status HTTP kepada Anda?
      Jika permintaan berhasil atau tidak (misalnya 200 bagus, 404 berarti tidak ditemukan).
    10. Apa pentingnya keadaan tanpa kewarganegaraan dalam arsitektur API?
      Setiap permintaan memiliki semua informasi yang dibutuhkan, sehingga membuat API lebih andal.
    11. Apa saja gaya arsitektur untuk membuat Web API?
      REST, SOAP, GraphQL
    12. Bandingkan perbedaan antara REST, SOAP, dan GraphQL.
      REST menggunakan sumber daya, SOAP menggunakan XML, GraphQL memungkinkan Anda meminta data spesifik.
    13. Apa prinsip desain API yang baik?
      Konsisten, mudah digunakan, terdokumentasi dengan baik, aman.
    14. Apa sajakah alat pengujian API yang umum?
      Postman, REST Assured, SoapUI
    15. Jelaskan langkah-langkah yang terlibat dalam pengujian API.
      Merencanakan, merancang pengujian, menyiapkan, menjalankan pengujian, melaporkan hasil.
    16. Bagaimana Anda melakukan Pengujian Beban API?
      Simulasikan banyak pengguna yang menggunakan API secara bersamaan.
    17. Apa itu contract testing?
      Memastikan layanan yang berbeda berkomunikasi satu sama lain dengan benar.
    18. Apa itu fuzz testing?
      Mengirim data aneh ke API untuk melihat apakah data tersebut rusak.
    19. Bagaimana pendekatan Anda dalam menguji API yang memiliki mekanisme autentikasi dan otorisasi yang kompleks?
      Uji secara menyeluruh berbagai peran dan izin pengguna.
    20. Bagaimana Anda menguji API yang belum sepenuhnya dikembangkan?
      Gunakan API tiruan atau stub untuk menyimulasikan bagian yang hilang.
    21. Bagaimana cara kami menguji keamanan API?
      Uji autentikasi, otorisasi, validasi input, dan pindai kerentanan.
    22. Apa sajakah kerentanan keamanan API yang umum?
      Serangan injeksi, otentikasi buruk, kebocoran data sensitif.

    Metrik dan Wawasan Utama untuk Para SQA

    Read More
    1. Defect Metrics
      1. Defect Density
        Jumlah cacat per modul atau fitur, membantu memprioritaskan area yang memerlukan perhatian.
      2. Defect Leakage
        Cacat yang ditemukan dalam produksi setelah pengujian, merupakan indikator utama dari cakupan yang terlewat.
      3. Defect Removal Efficiency (DRE)
        Persentase cacat yang dihilangkan selama pengujian versus cacat yang ditemukan setelah rilis.
      4. Insight
        DRE yang lebih tinggi menunjukkan pengujian yang efektif; targetkan di atas 90%.
    2. Test Coverage Metrics
      1. Requirements Coverage
        Persentase persyaratan yang dicakup oleh kasus uji.
      2. Code Coverage
        Persentase basis kode yang dieksekusi selama pengujian (misalnya, cakupan unit, fungsional, dan cabang).
      3. Insight
        Cakupan yang tinggi memastikan lebih sedikit kesenjangan dalam pengujian dan mengurangi kemungkinan bug yang tidak terdeteksi.
    3. Automation Metrics
      1. Automation Coverage
        Persentase kasus uji yang diotomatisasi dibandingkan dengan total kasus uji.
      2. Execution Time Savings
        Penghematan waktu menggunakan otomatisasi dibandingkan dengan pengujian manual.
      3. Pass/Fail Rate
        Rasio test case yang berhasil dan gagal dalam rangkaian otomatis.
      4. Insight
        ROI otomatisasi yang tinggi berarti rilis yang lebih cepat dan efisiensi biaya.
    4. Performance Metrics
      1. Response Time
        Waktu yang dibutuhkan sistem untuk merespons permintaan pada beban berbeda.
      2. Throughput
        Jumlah transaksi yang diproses per detik.
      3. Error Rate
        Frekuensi kesalahan yang terjadi selama pengujian kinerja.
      4. Insight
        Tingkat kesalahan yang rendah dan waktu respons yang optimal sangat penting untuk kepuasan pengguna.
    5. Testing Productivity Metrics
      1. Test Case Productivity
        Jumlah kasus uji yang dieksekusi per hari.
      2. Defects per Tester
        Jumlah cacat yang teridentifikasi per penguji, yang menunjukkan kontribusi individu.
      3. Insight
        Peningkatan produktivitas mencerminkan proses yang disederhanakan dan pemanfaatan sumber daya yang efektif.
    6. Customer Satisfaction Metrics
      1. Customer Reported Defects
        Jumlah masalah yang dilaporkan oleh pengguna pasca penerapan.
      2. Net Promoter Score (NPS)
        Mengukur kepuasan pengguna dan kemungkinan merekomendasikan produk.
      3. Insight
        Lebih sedikit cacat yang dilaporkan dan NPS yang lebih tinggi menandakan produk yang andal dan ramah pengguna.
    7. Project Metrics
      1. On-Time Delivery
        Persentase pencapaian pengujian yang dicapai sesuai jadwal.
      2. Test Budget Adherence
        Biaya yang dikeluarkan versus anggaran yang dialokasikan untuk kegiatan pengujian.
      3. Insight
        Manajemen sumber daya yang efisien menjaga pengujian tetap dalam cakupan dan tenggat waktu.

    Kesimpulan

    1. Ubah Wawasan menjadi Tindakan
    2. Pantau metrik secara teratur untuk mengidentifikasi tren dan mengoptimalkan proses.
    3. Gunakan kebocoran cacat dan DRE untuk meningkatkan cakupan pengujian.
    4. Manfaatkan metrik otomatisasi untuk menyeimbangkan upaya manual dan otomatis secara efektif.
    5. Analisis metrik kinerja untuk memastikan skalabilitas dan stabilitas dalam kondisi dunia nyata.
    6. Ini akan membantu tim kamu untuk memberikan solusi perangkat lunak berkualitas tinggi, hemat biaya, dan berpusat pada pengguna.

    Kepanjangan dari T.E.S.T.I.N.G

    Read More
    1. T - Test Cases
      Test Case yang terdefinisi dengan baik merupakan tulang punggung pengujian perangkat lunak yang efektif, memastikan setiap fungsi divalidasi dengan benar.
    2. E - Execution
      Jalannya kasus pengujian ini untuk memverifikasi apakah perangkat lunak berperilaku seperti yang diharapkan dalam kondisi dunia nyata.
    3. S - System Testing
      Pendekatan pengujian komprehensif yang memvalidasi seluruh sistem terhadap persyaratan yang ditentukan.
    4. T - Traceability
      Memastikan setiap persyaratan tercakup dalam satu atau lebih kasus pengujian, sehingga menjamin cakupan pengujian penuh.
    5. I - Integrasi
      Memverifikasi bahwa berbagai modul dan sistem bekerja sama dengan lancar sebagai bagian dari sistem yang lebih besar.
    6. N - Negative Testing
      Menguji bagaimana sistem menangani masukan yang tidak valid dan perilaku pengguna yang tidak terduga, untuk memastikan ketahanannya.
    7. G - GUI Testing
      Memastikan antarmuka pengguna grafis (GUI) berfungsi, responsif, dan ramah pengguna di berbagai perangkat.

    Ini hanya cara berpikir tentang pengujian, bukan singkatan formal.

    Mengapa Acceptance Criteria (AC) Penting dalam QA

    Read More

    Sebagai profesional QA, kami tahu bahwa Acceptance Criteria (AC) yang terdefinisi dengan baik adalah landasan keberhasilan pengujian. Tanpa hal ini, kita berisiko salah memahami persyaratan, menimbulkan cacat, dan ketidakselarasan dengan harapan klien.

    Inilah mengapa Acceptance Criteria sangat penting:

    1. Kejelasan
      Acceptance Criteria menghilangkan ambiguitas dengan mendefinisikan secara jelas apa arti "selesai" untuk sebuah fitur, mengurangi diskusi dan asumsi yang bolak-balik.
    2. Penyelarasan
      Ini menyelaraskan seluruh tim (QA, Pengembang, dan Bisnis) pada tujuan yang sama, memastikan kami menguji dengan tepat apa yang dibutuhkan.
    3. Hasil yang Dapat Diukur

    Apa itu UAT

    User Acceptance Testing (UAT) adalah fase terakhir dari proses pengujian perangkat lunak di mana pengguna akhir sebenarnya menguji sistem untuk memastikan sistem dapat menangani tugas-tugas yang diperlukan dalam skenario dunia nyata, sesuai dengan spesifikasi. Ini juga dikenal sebagai pengujian pengguna akhir, pengujian beta, atau pengujian aplikasi. Tujuan utama UAT adalah untuk memvalidasi bahwa perangkat lunak memenuhi kebutuhan bisnis dan siap untuk dirilis.

    Langkah-Langkah Penting dalam UAT:

    1. Perencanaan
      Identifikasi tujuan pengujian, pengguna, dan ruang lingkup pengujian.
    2. Test Case Desain
      Buat skenario dunia nyata berdasarkan bagaimana pengguna akhir akan menggunakan sistem.
    3. Eksekusi Tes
      Pengguna akhir menjalankan kasus pengujian, melaporkan masalah, dan memvalidasi aplikasi.
    4. Koleksi Umpan Balik
      Mengumpulkan umpan balik pengguna dan evaluasi jika ada perubahan yang diperlukan.
    5. Penandatanganan Terakhir
      Jika UAT berhasil maka perangkat lunak siap untuk dikerahkan.

    Manfaat dari UAT

    1. Memvalidasi Persyaratan Bisnis
      Memastikan bahwa produk akhir memenuhi tujuan dan persyaratan bisnis sebagaimana ditentukan oleh pemangku kepentingan. Hal ini menegaskan bahwa sistem berperilaku seperti yang diharapkan dalam skenario dunia nyata.
    2. Mengurangi Risiko Kegagalan
      Dengan melakukan uji penerimaan secara menyeluruh, organisasi dapat mengidentifikasi potensi masalah atau cacat sebelum produk dirilis ke pengguna akhir, sehingga mengurangi risiko kegagalan yang merugikan pasca peluncuran.
    3. Meningkatkan Kepercayaan Terhadap Produk
      Hal ini membangun kepercayaan di antara pemangku kepentingan, pelanggan, dan tim pengembangan bahwa produk siap untuk dijual, telah diuji dari sudut pandang pengguna.
    4. Menjamin Kepuasan Pelanggan
      Karena pengujian penerimaan berfokus pada verifikasi bahwa produk memenuhi kebutuhan pengguna, hal ini membantu memastikan bahwa produk akhir memberikan pengalaman pengguna yang positif, sehingga menghasilkan kepuasan pelanggan yang lebih tinggi.
    5. Deteksi Dini Masalah
      Pengujian penerimaan sering kali menangkap masalah yang mungkin terabaikan selama fase pengujian sebelumnya (seperti pengujian integrasi atau sistem), sehingga memungkinkan dilakukannya koreksi lebih awal sebelum penerapan.
    6. Meningkatkan Komunikasi
      Ini mendorong komunikasi yang lebih baik antara tim pengembangan dan pemangku kepentingan atau pengguna akhir. Kriteria penerimaan yang jelas memastikan bahwa semua pihak mempunyai pemahaman yang sama mengenai fungsi dan kinerja produk.
    7. Meminimalkan Pengerjaan Ulang
      Deteksi dini terhadap persyaratan yang tidak sepenuhnya dipahami atau dipenuhi mengurangi kemungkinan pengerjaan ulang ekstensif setelah rilis, sehingga menghemat waktu dan uang.
    8. Memfasilitasi Kepatuhan
      Pengujian penerimaan membantu memastikan bahwa produk mematuhi persyaratan peraturan, kewajiban kontrak, atau standar industri, terutama di bidang seperti layanan kesehatan, keuangan, atau layanan pemerintah.
    9. Bertindak sebagai Gerbang Kualitas Akhir
      Ini berfungsi sebagai pos pemeriksaan akhir sebelum produk diserahkan kepada klien atau pelanggan, memastikan bahwa produk bebas dari cacat kritis dan siap diproduksi.
    10. Mendukung Peningkatan Berkelanjutan
      Wawasan dari pengujian penerimaan dapat dimasukkan kembali ke dalam proses pengembangan. mengarah pada peningkatan dalam rilis produk di masa depan dan keselarasan yang lebih baik dengan harapan pengguna.

    Memahami Functional Testing

    Read More
    1. Smoke Testing
      Anggap saja ini sebagai pemeriksaan kesehatan cepat untuk perangkat lunak. Sebelum mendalami pengujian, kami memeriksa apakah aplikasi stabil dan siap untuk pengujian lebih lanjut.
    2. User Acceptance Testing (UAT)
      Ini adalah saat pengguna sebenarnya menguji aplikasi untuk memastikan aplikasi berfungsi seperti yang diharapkan dalam skenario dunia nyata. Jika mereka senang, perangkat lunak siap diluncurkan!
    3. Retesting
      Setelah memperbaiki bug, kami menjalankan kembali pengujian yang sama untuk mengonfirmasi bahwa masalah telah teratasi dan semuanya berfungsi dengan baik.
    4. System Testing
      Kami menguji seluruh aplikasi secara keseluruhan untuk memastikannya memenuhi persyaratan dan berfungsi dengan baik.
    5. Ad-hoc Testing
      Bayangkan menjelajahi aplikasi tanpa rencana, seperti yang dilakukan pengguna, untuk melihat apakah Anda dapat menemukan masalah yang tidak terduga.
    6. Regression Testing
      Setiap kali terjadi perubahan pada perangkat lunak (seperti memperbaiki bug atau menambahkan fitur baru), kami memeriksa apakah semuanya masih berfungsi sebagaimana mestinya.
    7. Database Testing
      Di sini, kami menguji bagaimana data disimpan, diambil, dan diperbarui dalam database aplikasi untuk memastikan keakuratan dan keamanannya.
    8. Integration Testing
      Ini tentang memeriksa bagaimana berbagai bagian aplikasi bekerja sama. Misalnya, apakah sistem login terhubung dengan benar ke database?
    9. Black Box Testing
      Anda tidak perlu mengetahui kode internal aplikasi. Periksa saja apakah masukan tersebut memberikan keluaran yang diharapkan, seperti menyelesaikan soal matematika tanpa mengetahui rumus di dalamnya.
    10. White Box Testing
      Kebalikan dari pengujian kotak hitam. Di sini, Anda menggali kode dan menguji struktur dan logikanya untuk memastikannya berjalan dengan benar.
    11. Recovery Testing
      Kami menguji seberapa baik sistem dapat bangkit kembali setelah mengalami crash atau kegagalan. Misalnya, apa yang terjadi jika aplikasi tiba-tiba mati?
    12. Functional Testing
      Ini berarti menguji apakah perangkat lunak melakukan apa yang seharusnya dilakukan berdasarkan persyaratan.
    13. Grey Box Testing
      Campuran pengujian kotak hitam dan putih. Anda tahu sedikit tentang struktur internal tetapi tetap fokus pada cara aplikasi berfungsi secara keseluruhan.
    14. Component Testing
      Setiap bagian (atau komponen) perangkat lunak diuji secara individual untuk memastikan perangkat lunak berfungsi dengan sempurna.
    15. Unit Testing
      Bagian terkecil dari kode, seperti fungsi atau modul tunggal, diuji untuk memeriksa apakah berfungsi dengan benar.
    16. Acceptance Testing
      Langkah terakhir di mana kami memeriksa apakah perangkat lunak memenuhi semua kebutuhan bisnis dan pengguna sebelum siap diluncurkan.

    Secara sederhana: Pengujian fungsional seperti memastikan setiap fitur perangkat lunak berfungsi dengan benar, baik itu tombol kecil atau keseluruhan aplikasi!

    1. GET
      Mengambil data dari server. Biasa digunakan untuk mengambil sumber daya tanpa mengubahnya.
    2. POST

      Menggaungkan permintaan yang diterima untuk tujuan diagnostik atau debugging.
    9. CONNECT
      Membuat terowongan untuk komunikasi aman antara klien dan server, sering kali digunakan untuk HTTPS.

    Epic

    1. Definisi
      Epic adalah kumpulan karya besar yang dapat dipecah menjadi bagian-bagian kecil yang disebut cerita pengguna. Ini mewakili fitur besar atau inisiatif signifikan yang akan dikerjakan tim dalam beberapa sprint.
    2. Tujuan
      Epik membantu menangkap tujuan atau sasaran yang luas. Seringkali mereka mewakili fungsionalitas yang terlalu besar untuk diselesaikan dalam satu sprint.
    3. Contoh
      Terapkan proses checkout untuk situs web e-commerce.

    User Story

    1. Definisi
      User Story adalah deskripsi kecil dan ringkas tentang fitur atau fungsi dari sudut pandang pengguna akhir. Ini mendefinisikan persyaratan spesifik yang dapat diselesaikan dalam satu sprint.
    2. Tujuan
      Berfokus pada apa yang diinginkan pengguna dan alasannya. Mereka membantu tim memahami bagaimana fitur tersebut akan memberikan nilai kepada pengguna.
    3. Struktur
      Biasanya ditulis dalam format: Sebagai [tipe pengguna], saya ingin [beberapa tujuan] sehingga [beberapa alasan/manfaat].
    4. Contoh
      Sebagai pelanggan, saya ingin menambahkan item ke keranjang saya sehingga saya dapat meninjaunya sebelum membeli.

    Task

    1. Definisi
      Tugas adalah unit pekerjaan tertentu yang perlu dilakukan untuk menyelesaikan cerita pengguna. Tugas memecah cerita pengguna menjadi item yang dapat ditindaklanjuti untuk pengembang, penguji, atau anggota tim lainnya.
    2. Tujuan
      Tugas mewakili langkah-langkah implementasi, pengkodean, pengujian, atau perancangan pekerjaan yang diperlukan untuk mencapai tujuan cerita pengguna.
    3. Contoh
      "Buat skema database untuk item keranjang belanja" atau "Tulis pengujian unit untuk fungsionalitas tambahkan ke keranjang."

    Hubungan antara Epic, User Story, dan Task

    1. Epic dipecah menjadi cerita pengguna
    2. User story selanjutnya dipecah menjadi tasks yang dapat dikerjakan tim untuk mengimplementasikan cerita tersebut.
    3. Sebuah Epic mungkin berisi beberapa User Stories, dan setiap user story mungkin memiliki beberapa task yang terkait dengannya.

    Apa itu QA Manual?

    QA Manual melibatkan penguji manusia yang berinteraksi dengan perangkat lunak untuk memastikan perangkat lunak tersebut memenuhi kebutuhan bisnis dan pengguna. Penguji mengidentifikasi cacat, memvalidasi fitur, dan memastikan aplikasi siap diterapkan.

    Poin Penting tentang QA Manual

    1. Berpusat pada Manusia
      Mengandalkan intuisi dan kreativitas untuk menemukan masalah UX/UI.
    2. Pengujian secara Nyata
      Mensimulasikan skenario pengguna untuk mengungkap masalah praktis.
    3. Sempurna untuk Fitur Baru
      Ideal untuk menguji fungsionalitas baru sebelum otomatisasi.
    4. Didorong oleh Umpan Balik
      Mendokumentasikan bug dan inkonsistensi yang harus diperbaiki oleh pengembang.

    Mengapa QA Manual itu Penting?

    QA manual memainkan peran penting dalam memastikan kualitas perangkat lunak dengan menangani aspek-aspek yang seringkali tidak dapat dilakukan oleh otomatisasi.

    Alasan Utama QA Manual Penting

    1. Berpusat pada Pengguna
      Memastikan perangkat lunak intuitif dan ramah pengguna.
    2. Mendeteksi Masalah
      Menemukan bug yang mungkin terlewatkan oleh otomatisasi.
    3. Sangat Mudah Beradaptasi
      Ideal untuk menjelajahi fitur baru atau dinamis.
    4. Hemat Biaya
      Praktis untuk proyek kecil atau jangka pendek.
    5. Penting untuk UX/UI
      Validasi desain dan interaksi pengguna.

    Tanggung Jawab Inti QA Manual

    1. Analisis Persyaratan
      Memahami kebutuhan bisnis dan pengguna.
    2. Menulis Test Case
      Buat skenario terperinci untuk validasi.
    3. Jalankan Tes
      Lakukan pengujian langkah demi langkah untuk menemukan bug.
    4. Catat Bug
      Laporkan masalah dan berkolaborasi dalam perbaikan.
    5. Tes Ulang dan Verifikasi
      Pastikan bug teratasi dan tidak ada bug baru yang muncul.
    6. Hasil Dokumen
      Meringkas hasil pengujian dan kualitas perangkat lunak.

    Keterampilan yang Diperlukan untuk QA Manual

    1. Perhatian terhadap Detil
      Temukan cacat kecil yang mungkin terlewatkan oleh orang lain.
    2. Berpikir Kritis
      Analisis kasus edge dan skenario tak terduga.
    3. Keterampilan Komunikasi
      Laporkan temuan dengan jelas kepada tim.

    Proses QA Manual

    1. Memahami Requirements
      Analisis persyaratan dan tujuan proyek.
    2. Test Case Desain
      Buat skenario pengujian yang terperinci dan komprehensif.
    3. Catat Bug
      Dokumentasikan masalah dengan langkah-langkah yang jelas untuk mereproduksinya.
    4. Verifikasi Perbaikan
      Uji ulang masalah yang terselesaikan dan validasi perbaikan.
    5. Hasil Dokumen
      Ringkas temuan dan berikan umpan balik kepada tim pengembangan.

    Apa tantangan bagi QA Manual?

    1. Tenggat waktu yang ketat dan waktu pengujian yang terbatas.
    2. Dokumentasi tidak lengkap atau tidak jelas.
    3. Menangani kasus tepi dalam sistem yang kompleks.

    Kesimpulan

    1. QA Manual memastikan perangkat lunak dapat diandalkan, mudah digunakan, dan bebas bug.
    2. Ini melengkapi pengujian otomatis untuk proses QA yang menyeluruh.

    Siklus Hidup Bug (Bug Life Cycle) adalah proses sistematis yang memandu suatu cacat mulai dari awal hingga penyelesaian akhir. Berikut rincian tahapan utamanya:

    1. New
      Bug diidentifikasi dan dicatat.
    2. Assigned
      Bug tersebut ditugaskan ke pengembang untuk dianalisis dan diselesaikan.
    3. Open
      Pengembang mulai mengerjakan perbaikannya.
    4. Fixed
      Pengembang menerapkan solusi dan menandai bug sebagai telah diperbaiki.
    5. Retest
      Tim QA memverifikasi perbaikan di lingkungan tertentu.
    6. Verified
      Perbaikan telah dikonfirmasi, dan bug telah ditutup.
    7. Reopen
      Jika bug masih ada, maka akan dibuka kembali untuk penyelidikan lebih lanjut.
    8. Closed
      Bug telah teratasi secara permanen.
    9. Deferred
      Bug dengan prioritas rendah mungkin ditunda.
    10. Rejected
      Bug yang tidak valid atau tidak dapat direproduksi akan dihilangkan.

    1. 201 Created
      Sumber daya baru berhasil dibuat. Sering terlihat dengan permintaan POST.
    2. 204 No Content
      Permintaan berhasil, namun tidak ada konten dalam responsnya. Umum dengan permintaan DELETE.

      Menunjukkan bahwa sumber daya tidak berubah sejak permintaan terakhir. Berguna untuk menyimpan cache.

    1. Unit Testing
      Mengisolasi unit kode individual untuk memastikan unit tersebut berfungsi sesuai harapan.

    2. Integration Testing

    4. Acceptance Testing

      Anggap saja sebagai upacara pembukaan gedung Anda.

    1. Performance Testing
      Menilai kecepatan, daya tanggap, dan skalabilitas di bawah beban yang berbeda.
      Bayangkan menguji berapa banyak orang yang dapat ditampung bangunan Anda dengan aman.

    1. Regression Testing
      Memastikan perubahan baru tidak merusak fungsi yang sudah ada.
      Bayangkan memeriksa retakan pada bangunan Anda setelah renovasi.
    2. Smoke Testing
      Pemeriksaan kewarasan cepat untuk memastikan fungsionalitas dasar sebelum pengujian lebih lanjut.
      Pikirkan untuk menyalakan lampu dan memeriksa fungsionalitas sistem dasar sebelum pemeriksaan lebih dalam.
    3. Exploratory Testing
      Pengujian kreatif dan tidak terstruktur untuk mengungkap masalah yang tidak terduga.
      Mirip dengan detektif yang mencari petunjuk tersembunyi di gedung Anda.

    Penjelasan Metode Otentikasi REST API

    Read More
    1. Basic Authentication
      Bagaimana cara kerjanya: Mengirimkan nama pengguna dan kata sandi yang dikodekan dalam base64 dengan setiap permintaan.
      Pros: Sederhana, mudah diterapkan, didukung secara luas.
      Cons: Kredensial dikirim pada setiap panggilan, sehingga rentan kecuali digunakan dengan HTTPS.
      Terbaik untuk: API internal atau lingkungan pengembangan, tetapi tidak direkomendasikan untuk produksi.
    2. Token Authentication
      Bagaimana cara kerjanya: Client mengautentikasi satu kali, mendapatkan token, dan menggunakannya untuk permintaan berikutnya.
      Pros: Lebih aman dibandingkan Otentikasi Dasar, token dapat dicabut, mengurangi beban server.
      Cons: Memerlukan manajemen token, menimbulkan risiko jika token disusupi.
      Terbaik untuk: Web, mobile apps, SPAs (Single Page Applications).
    3. OAuth Authentication
      Bagaimana cara kerjanya: Mengizinkan aplikasi pihak ketiga mengakses sumber daya tanpa membagikan kata sandi. Melibatkan beberapa langkah (permintaan, pemberian, akses, penyegaran token).
      Pros: Sangat aman, ideal untuk integrasi pihak ketiga, menawarkan kontrol akses yang sangat baik.
      Cons: Rumit untuk diterapkan, mungkin berlebihan untuk API yang lebih sederhana.
      Terbaik untuk: API yang terintegrasi dengan beberapa layanan atau mengizinkan akses pihak ketiga.
    4. API Key Authentication
      Bagaimana cara kerjanya: Client menyertakan kunci API dalam header atau parameter kueri.
      Pros: Mudah diimplementasikan dan digunakan, berguna untuk melacak penggunaan API.
      Cons: Kurang aman jika kunci terekspos, kontrol akses terbatas.
      Terbaik untuk: API Publik, layanan yang berfokus pada pengembang, atau saat pelacakan penggunaan API itu penting.

    Poin Penting

    1. Selalu gunakan HTTPS untuk mengenkripsi data saat transit, apa pun metodenya.
    2. Keseimbangan antara keamanan dan kemudahan penggunaan.
    3. Menerapkan kebijakan manajemen dan rotasi token/kunci yang tepat.
    4. Tetap terinformasi tentang praktik terbaik keamanan dan standar yang terus berkembang.

    Apa itu Bug Priority

    Prioritas bug mengacu pada seberapa mendesak suatu bug harus diperbaiki dan dihilangkan dari situs web atau aplikasi.

    Tingkat dari Bug Priority:

    1. Low
      Bug dapat diperbaiki di kemudian hari. Bug lain yang lebih parah akan diprioritaskan.
    2. Medium
      Bug dapat diperbaiki dalam proses pengembangan dan pengujian biasa.
    3. High
      Bug harus diatasi segera setelah berdampak buruk pada sistem dan menjadikannya tidak dapat digunakan sampai diselesaikan.

      Pengujian antar muka pengguna (UI) dan pengalaman pengguna (UX) yang mencakup evaluasi visual dan memerlukan pengamatan manusia.
    3. Ad-hoc Testing
      Tes spontan yang dilakukan tanpa mengikuti persyaratan atau dokumentasi dan ditujukan untuk mengungkap bug yang tidak terduga.

    Kapan harus menggunakan Automated Testing?

    1. Smoke Testing
      Memeriksa fungsionalitas inti produk/aplikasi. Biasanya agak standar dan dapat digunakan kembali.
    2. Regression Testing
      Menguji ulang fungsionalitas yang ada setelah pembaharuan ke aplikasi itu sendiri atau sistem operasi dan dirancang untuk mengetahui apakah ada masalah yang terjadi.

      Pengujian tegangan untuk mengetahui dimana letak batas beban atau masalah ketika aplikasi error saat digunakan oleh banyak user.
    5. Repeated Execution
      Beberapa tes dijalankan berulang kali pada titik pengembangan tertentu atau tes standar pada aplikasi yang berbeda.

    1. Apa itu SQL String Functions?
      Fungsi string SQL memungkinkan manipulasi teks, transformasi, dan analisis dalam kueri database untuk pengelolaan data yang lebih efektif. Contoh:

      SELECT UPPER('hello world'); -- Output: HELLO WORLD

    2. Jenis String Functions
      1. Fungsi Konversi
        Ubah huruf besar/kecil string menjadi huruf besar, huruf kecil, atau huruf besar pada huruf pertama. Contoh:
        1. Mengembalikan tabel
          SELECT UPPER('sql functions'); -- Output: SQL FUNCTIONS
        2. Mengonversi teks menjadi huruf kecil
          SELECT LOWER('SQL Functions'); -- Output: sql functions
      2. Fungsi Panjang dan Substring
        Hitung jumlah karakter atau ekstrak bagian string.
        1. Menghitung karakter
          SELECT LEN('Database'); -- Output: 8
        2. Ekstrak substring
          SELECT SUBSTRING('Database', 1, 4); -- Output: Data
        3. Menemukan posisi substring
          SELECT CHARINDEX('base', 'Database'); -- Output: 5
      3. Fungsi Penggabungan
        Gabungkan beberapa string menjadi satu.
        1. CONCAT()
          SELECT CONCAT('SQL', ' ' 'Functions'); -- Output: SQL Functions
        2. Menggunakan + (SQL Server)
          SELECT 'SQL' + ' Functions'; -- Output: SQL Functions
        3. Menggunakan || (PostgreSQL, Oracle)
          SELECT 'SQL' || ' Functions'; -- Output: SQL Functions
      4. Fungsi Pencarian dan Penggantian
        Temukan atau ganti teks tertentu.
        1. Menemukan posisi substring
          SELECT CHARINDEX('SQL', 'Learn SQL Functions'); -- Output: 7
        2. Mengembalikan tanggal dan waktu saat ini
          SELECT REPLACE('SQL is awesome', 'awesome', 'powerful'); -- Output: SQL is powerful
      5. Fungsi Pemangkasan
        Hapus spasi atau karakter yang tidak diinginkan dari string.
        1. Menghapus spasi awal/akhir
          SELECT LTRIM(' SQL Functions'); -- Output: SQL Functions
          SELECT RTRIM('SQL Functions '); -- Output: SQL Functions
        2. Menghapus spasi awal dan akhir atau karakter tertentu
          SELECT TRIM(' SQL Functions '); -- Output: SQL Functions
      6. Fungsi Perbandingan String
        Membandingkan string untuk persamaan atau kecocokan parsial.
        1. Menemukan sebagian kecocokan
          SELECT * FROM employees WHERE name LIKE 'J%'; -- Output: Names starting with J
        2. Membandingkan string
          SELECT STRCMP('SQL', 'sql'); -- Output: -1 (case-sensitive)
      7. Mengubah String menjadi Angka
        Menyederhanakan fungsi bernilai tabel dengan mendefinisikannya dalam satu kueri.

        Mengonversi string menjadi angka.
        SELECT CAST('123', AS INT); -- Output: 123
        SELECT CONVERT(INT, '123'); -- Output: 123

      8. Kasus Penggunaan Umum untuk Fungsi String SQL
        1. Standarisasi Teks
          SELECT UPPER(name) FROM employees; -- Ubah semua nama menjadi huruf besar
        2. Filtering Data
          SELECT * FROM customers WHERE email LIKE '%@gmail.com';
        3. Mengekstraksi Informasi
          SELECT SUBSTRING(phone, 1, 3) AS area_code FROM customers;
      9. Menggabungkan Beberapa Fungsi String
        Gunakan beberapa fungsi string untuk transformasi tingkat lanjut.

        SELECT UPPER(SUBSTRING('Learn SQL Functions', 7, 3,)); -- Output: SQL

      10. Fungsi String dalam Dialek SQL yang Berbeda
        Menggunakan ALTER FUNCTION untuk memperbarui fungsi yang ada.
        1. Digabungkan dengan pemisah (MySQL)
          SELECT CONCAT_WS('-', 'SQL', 'Functions'); -- Output: SQL-Functions
        2. Menemukan substring (PostgreSQL)
          SELECT POSITION('SQL' IN 'Learn SQL Functions'); -- Output: 7
        3. Mengembalikan posisi suatu pola (MySQL)
          SELECT PATINDEX('%SQL%', 'Learn SQL Functions'); -- Output: 7
      11. Tips Performa untuk Fungsi String
        1. Avoid using functions on indexed columns in WHERE clauses.
          1. Tidak efisien
            SELECT * FROM employees WHERE UPPER(name) = 'JOHN';
          2. Efisien
            SELECT * FROM employees WHERE name = 'John';
        2. Pra-proses teks di luar kueri jika memungkinkan.
    3. Penerapan
      Gunakan fungsi string untuk membakukan teks, mengekstrak informasi spesifik, memfilter data, atau menggabungkan dan memanipulasi data tekstual secara efisien.
    4. Praktik Terbaik
      Hindari menerapkan fungsi ke kolom yang diindeks, melakukan praproses teks jika memungkinkan, dan menggabungkan beberapa fungsi untuk transformasi tingkat lanjut.

        Daftar prioritas yang berisi semua fitur, penyempurnaan, perbaikan bug, dan pekerjaan teknis yang diperlukan untuk menyempurnakan produk. Product Owner bertanggung jawab untuk mengelola dan memelihara simpanan.
      2. Sprint Backlog
        Subset item dari product backlog yang dipilih untuk sprint. Tim pengembangan berkomitmen untuk mengirimkan barang-barang ini selama sprint.
      3. Increment
        Jumlah semua item product backlog yang diselesaikan selama sprint, dikombinasikan dengan penambahan semua sprint sebelumnya. Ini mewakili keadaan produk saat ini dan harus dalam kondisi yang dapat digunakan.
    3. Events
      1. Sprint
        Inti dari Scrum, periode waktu tertentu (biasanya 2-4 minggu) di mana tim bekerja untuk menyelesaikan serangkaian item product backlog. Tujuan dari setiap sprint adalah untuk menghasilkan peningkatan kerja produk.
      2. Sprint Planning
        Pada awal setiap sprint, tim Scrum mengadakan rapat perencanaan sprint untuk menentukan tujuan sprint dan menentukan item product backlog mana yang akan diselesaikan selama sprint.
      3. Daily Scrum / Stand-up
        Pertemuan berdurasi 15 menit yang diadakan setiap hari di mana anggota tim pengembang mendiskusikan kemajuan mereka, apa yang mereka rencanakan untuk dilakukan, dan hambatan apa pun yang mereka hadapi. Ini membantu menjaga tim tetap pada jalurnya.
      4. Sprint Review
        Di akhir sprint, tim mendemonstrasikan peningkatan yang telah diselesaikan kepada pemangku kepentingan. Tujuannya adalah untuk mendapatkan umpan balik dan menyesuaikan product backlog jika diperlukan.
      5. Sprint Retrospective
        Pertemuan dimana tim Scrum merefleksikan sprint yang baru saja selesai, mendiskusikan apa yang berjalan dengan baik, apa yang tidak, dan bagaimana proses tersebut dapat ditingkatkan pada sprint berikutnya.
    4. Scrum Artifacts and Transparency
      Scrum menekankan visibilitas kemajuan dan permasalahan. Product backlog dan sprint backlog membuat kemajuan menjadi transparan, sedangkan Daily Scrum membantu inspeksi berkelanjutan.
    5. Bagaimana Cara Kerja Scrum?
      1. Scrum Beroperasi di Sprint
        Iterasi pekerjaan yang singkat dan terbatas waktu (biasanya 2-4 minggu). Setiap sprint mengikuti siklus yang teratur, dimulai dengan perencanaan sprint, di mana tim memilih pekerjaan yang harus dilakukan, dan diakhiri dengan tinjauan sprint dan retrospektif, di mana tim merefleksikan kemajuan mereka dan mendiskusikan perbaikan.
      2. Incremental Delivery
        Scrum mempromosikan pemberian produk dalam jumlah kecil dan dapat digunakan pada akhir setiap sprint, sehingga memungkinkan tim untuk memberikan nilai lebih awal dan sering.
      3. Continuous Improvement
        Melalui retrospektif, tim terus meningkatkan proses, meningkatkan efisiensi dari waktu ke waktu.
      4. Flexibility and Adaptability
        Product backlog terus disempurnakan, memungkinkan penyesuaian cakupan berdasarkan umpan balik dan perubahan persyaratan.
    6. Manfaat Menerapkan Scrum
      1. Improved Collaboration
        Scrum mendorong kolaborasi yang erat antara pemilik produk, master Scrum, dan tim pengembangan, sehingga mendorong komunikasi dan keselarasan yang lebih baik.
      2. Increased Flexibility
        Tim dapat dengan cepat merespons perubahan permintaan pasar atau kebutuhan pelanggan melalui pengembangan berulang.
      3. Faster Time to Market
        Dengan memberikan peningkatan yang dapat digunakan di akhir setiap sprint, Scrum memungkinkan bisnis untuk merilis fitur dan produk dengan lebih cepat.
      4. Higher Product Quality
        Pengujian berkelanjutan, umpan balik, dan perbaikan berulang-ulang akan menghasilkan produk berkualitas lebih tinggi.
      5. Better Risk Management
        Meninjau sprint dan retrospektif secara rutin dapat membantu mengidentifikasi dan mengatasi risiko sejak dini.

    SQL Constraint: Memastikan Integritas Data di Database Anda

    Read More
    1. Apa itu SQL Constraint
      SQL Constraint adalah aturan yang diterapkan pada kolom tabel untuk memastikan integritas data dengan menerapkan akurasi, konsistensi, dan keandalan. Contoh:

      CREATE TABLE employees ( emp_id INT PRIMARY KEY, email VARCHAR (50) UNIQUE );
      emp_id harus unik, email tidak boleh berulang

    2. Jenis SQL Constraint
      SQL menyediakan beberapa tipe batasan untuk menerapkan aturan yang berbeda.
      1. NOT NULL
        Memastikan kolom tidak boleh memiliki nilai null (kosong).. Contoh:

        CREATE TABLE customers ( customer_id INT NOT NULL, name VARCHAR(50) NOT NULL );
        Dari kasus diatas, menerapkan bidang wajib seperti customer name atau customer id.

      2. UNIQUE
        Mencegah nilai duplikat dalam kolom. Contoh:

        CREATE TABLE users ( email VARCHAR(100) UNIQUE );
        Dari kasus diatas, menjamin pengidentifikasi unik seperti alamat email atau nama pengguna.

      3. PRIMARY KEY
        Secara unik mengidentifikasi setiap catatan dalam sebuah tabel. Gabungan NOT NULL dan UNIQUE Constraint. Contoh:

        CREATE TABLE employees ( emp_id INT PRIMARY KEY, name VARCHAR(50) );
        Dari kasus diatas, mengidentifikasi baris secara unik, memastikan tidak ada duplikat.

      4. FOREIGN KEY
        Mempertahankan integritas referensial dengan menghubungkan dua tabel. Contoh:

        CREATE TABLE orders ( order_id INT PRIMARY KEY, customer_id INT, FOREIGN KEY (customer_id) REFERENCES customers(customer_id) );
        Dari kasus diatas, memastikan bahwa customer_id dalam orders ada di tabel customers.

      5. CHECK
        Membatasi nilai yang diperbolehkan dalam kolom berdasarkan suatu kondisi. Contoh:

        CREATE TABLE products ( product_id INT PRIMARY KEY, price DECIMAL(10, 2) CHECK (price > 0) );
        Dari kasus diatas, menegakkan aturan bisnis seperti penetapan harga positif.

      6. DEFAULT
        Menetapkan nilai default untuk kolom jika tidak ada nilai yang diberikan selama penyisipan. Contoh:

        CREATE TABLE accounts ( account_id INT PRIMARY KEY, balance DECIMAL(10, 2) DEFAULT 0.00 );
        Dari kasus diatas, secara otomatis menginisialisasi kolom seperti balance atau timestamp.

      7. DEFAULT
        Menetapkan nilai default untuk kolom jika tidak ada nilai yang diberikan selama penyisipan. Contoh:

        CREATE TABLE accounts ( account_id INT PRIMARY KEY, balance DECIMAL(10, 2) DEFAULT 0.00 );
        Dari kasus diatas, secara otomatis menginisialisasi kolom seperti balance atau timestamp.

      8. COMBINING
        Kamu dapat menerapkan beberapa batasan pada satu kolom untuk aturan yang lebih ketat. Contoh:

        CREATE TABLE inventory ( item_id INT PRIMARY KEY, stock INT NOT NULL CHECK (stock > 0) );
        Dari kasus diatas, memastikan item_id unik dan nilai stok valid.

      9. DROPPING
        Menghapus batasan yang tidak perlu untuk memperbarui aturan tabel. Contoh:
        1. Menghilangkan batasan UNIQUE
          ALTER TABLE employees DROP CONSTRAINT unique_email;
        2. Menghilangkan batasan FOREIGN KEY
          ALTER TABLE orders DROP CONSTRAINT fk_customer_id;
      10. Menambahkan Constraint pada Tabel yang Ada
        Kamu dapat menambahkan batasan ke tabel yang sudah ada menggunakan ALTER TABLE. Contoh:
        1. Menambahkan batasan UNIQUE
          ALTER TABLE employees ADD CONSTRAINT unique_email UNIQUE (email);
        2. Menambahkan batasan CHECK
          ALTER TABLE products ADD CONSTRAINT check_price CHECK (price > 0);
    3. Mengapa harus menggunakan SQL Constraint?
      1. Menjamin keakuratan dan validitas data.
      2. Mencegah entri data yang tidak valid atau tidak konsisten.
      3. Sederhanakan logika validasi dalam aplikasi.
    4. Cara terbaik untuk Constraint
      1. Desain untuk kejelasan
        Gunakan batasan untuk mencerminkan aturan dunia nyata.
      2. Kombinasikan batasan
        Memanfaatkan berbagai batasan untuk validasi yang kuat.
      3. Uji secara menyeluruh
        Pastikan batasan berfungsi sebagaimana mestinya sebelum penerapan.

      CREATE TABLE transactions ( txn_id INT PRIMARY KEY, amount DECIMAL(10, 2) NOT NULL CHECK (amount > 0) );

    5. Mengecek Constraint pada SQL
      Kueri database Anda untuk melihat batasan yang diterapkan pada tabel.
      1. MySQL
        SHOW CREATE TABLE employees; -- Tampilkan semua batasan pada tabel
      2. PostgreSQL
        SELECT conname, contype FROM pg_constraint WHERE conrelid = 'employees' ::regclass;
    6. Batas dari SQL Constraint
      1. Batasan tidak dapat menangani validasi yang rumit (misalnya logika lintas kolom).
      2. Tidak dapat menggantikan logika aplikasi spesifik bisnis.
      3. Kinerja yang berlebihan untuk tabel besar dengan perubahan yang sering.
    7. Kapan harus menghindari Constraint
      1. Validasi data yang sudah diterapkan pada aplikasi.
      2. Fleksibilitas diperlukan untuk data sementara atau uji.

      Memasukkan data pengujian yang tidak konsisten mungkin memerlukan penonaktifan batasan:
      SET FOREIGN_KEY_CHECKS = 0;
      -- Masukkan data uji
      SET FOREIGN_KEY_CHECKS = 1;

    8. Penggunaan Constraint Tingkat Lanjut
      1. Menerapkan batasan pada sub-kumpulan data (Partial Constraint)
        CREATE UNIQUE INDEX idx_unique_active ON users(email) WHERE is_active = 1;
      2. Dynamic Defaults (PostgreSQL)
        CREATE TABLE logs ( log_id SERIAL PRIMARY KEY, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );

    29. Rayakan Kemenangan Kecil
      Akui pencapaian Anda dan terus memotivasi diri sendiri.
